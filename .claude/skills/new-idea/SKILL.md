# /new-idea - アイデア壁打ち・企画書作成スキル

## 概要

ユーザーのアイデアについて対話形式で壁打ちを行い、率直なフィードバックを通じてブラッシュアップする。
対話が十分に深まった段階で、企画書・壁打ち記録・参考資料をドキュメントとして生成し、GitHub Issue に登録する。

**重要**: このスキルは自動パイプライン型ではなく**対話駆動型**。ユーザーとのやり取りを通じて段階的に進める。

## ワークフロー

以下のステップを順番に実行する。Step 2〜3 はユーザーとの対話を挟むため、ユーザーの応答を待ってから次に進むこと。

### Step 1: プロフィール読み込みと準備

1. Read ツールで `/home/a/00.knowledge-hub/PROFILE.md` を読み込み、スキルセット・興味領域・制約を把握する
2. Glob ツールで `02.Ideas/**/*` を確認し、既存アイデアのリストを取得する（重複チェック用）

### Step 2: アイデアのヒアリング開始

ユーザーに以下のように話しかけて対話を開始する:

> どんなアイデアについて話しましょうか？
> ざっくりした構想でも、具体的な仕様でも、どちらでも大丈夫です。

ユーザーの回答を受けて、以下の観点で**最大3問まで**追加質問を行う:
- **What**: 具体的に何を作りたいのか
- **Why**: なぜそれが必要なのか・どんな課題を解決するのか
- **Who**: 誰が使うのか（自分？他の人？）
- **差別化**: 既存の類似サービスとの違いは何か

**注意**: 1回の応答で質問は最大3問まで。質問攻めにしない。

### Step 3: 深掘りと率直なフィードバック（2〜3ラウンド）

ユーザーの回答を受けて、`references/evaluation-criteria.md` の5軸に基づいてフィードバックを行う。

#### フィードバックの姿勢

- **良い物には良い、悪い物には悪いとはっきり言う**
- 良い点は素直に認め、具体的に何が良いかを伝える
- 問題がある部分には率直に指摘する（曖昧にしない）
- 指摘には**必ず代替案・改善案を添える**
- 目的はアイデアを潰すことではなく、より良いアイデアにブラッシュアップすること

#### フィードバックの例

良い例:
> 「自分が毎日使いたい」というモチベーションは強力です。Dogfooding の観点で高評価です。
> 一方で、技術的にはリアルタイム同期が必要になり、WebSocket の実装経験がないと工数が膨らむリスクがあります。
> 代替案として、まずはポーリング方式で MVP を作り、後からリアルタイム化する段階的アプローチはどうでしょう？

悪い例:
> いいアイデアですね！（具体性がない肯定）
> 難しいと思います。（代替案がない否定）

#### ラウンドの進め方

- 各ラウンドで1〜2個の論点に絞ってフィードバックする
- ユーザーの反応を見て、次のラウンドのテーマを決める
- 2〜3ラウンドで情報が揃ったら、以下のように提案する:

> アイデアの方向性が見えてきましたね。企画書としてまとめましょうか？
> 他に議論したいポイントがあれば、もう少し話しましょう。

### Step 4: カテゴリ分類

AskUserQuestion ツールを使って、アイデアのカテゴリをユーザーに確認する。

カテゴリ選択肢:
- **技術系** (`type:tech`): 開発ツール・ライブラリ・技術的なプロジェクト
- **事業** (`type:business`): 収益化を目指すサービス・プロダクト
- **趣味・プライベート** (`type:hobby`): 個人的な趣味や生活改善のためのプロジェクト
- **その他** (`type:other`): 上記に該当しないもの

### Step 5: 企画書・記録の生成

以下の3ファイルを生成する。出力先は `02.Ideas/YYYY-MM-DD_{title}/` とする。

#### 5-1. overview.md（企画書）

`references/overview-template.md` のテンプレートに従って生成する。

- 5W1H を全て埋める
- `references/evaluation-criteria.md` の5軸で実現可能性を評価する
- リスクと懸念事項を最低1つは記載する
- MVP 定義は可能な限り小さいスコープにする
- 技術スタックは PROFILE.md のスキルセットを考慮して提案する
- ネクストアクションは具体的かつ着手可能な粒度で記載する

#### 5-2. discussion.md（壁打ち記録）

`references/discussion-template.md` のテンプレートに従って生成する。

- Step 2〜3 の対話を要約して記録する
- 議論のまとめセクションを充実させる（アイデアの変遷、主要論点、フィードバック）
- 未解決の課題があれば記載する

#### 5-3. references.md（参考資料）

以下の形式で生成する:

```markdown
# 参考資料: {アイデアタイトル}

> 作成日: {YYYY-MM-DD}

## 類似サービス・競合

- [{サービス名}]({URL}): {概要と差別化ポイント}

## 参考記事・ドキュメント

- [{記事タイトル}]({URL}): {参考になるポイント}

## 使えそうな技術・ライブラリ

- [{技術名}]({URL}): {用途}
```

- 対話の中で出てきた参考情報を整理する
- 情報が不足している場合は WebSearch ツールで補完する
- 最低でも「類似サービス・競合」セクションは埋める

### Step 6: ユーザー確認

生成した企画書（overview.md）の内容をユーザーに表示し、確認を取る。

> 企画書を作成しました。内容を確認してください。
> 修正したい箇所があればお知らせください。

修正の依頼があれば対応する。なければ次のステップに進む。

### Step 7: GitHub Issue 登録

ToolSearch ツールで `mcp__github__create_issue` を読み込み、以下の内容で Issue を作成する。

- **owner**: `silmo-yokohama`
- **repo**: `knowledge-hub`
- **title**: `[アイデア] {アイデアタイトル}`
- **labels**: Step 4 で選択したカテゴリラベル + `status:ideation`
  - 技術系 → `type:tech`, `status:ideation`
  - 事業 → `type:business`, `status:ideation`
  - 趣味・プライベート → `type:hobby`, `status:ideation`
  - その他 → `type:other`, `status:ideation`
- **body**: 以下の形式で作成する

```markdown
## 概要

{アイデアの概要を2〜3文で}

## 課題

{解決したい課題}

## ソリューション

{提案するソリューション}

## MVP

{MVP に含める最小限の機能をチェックリストで}

- [ ] {機能1}
- [ ] {機能2}
- [ ] {機能3}

## 実現可能性

| 軸 | 評価 |
|----|------|
| 技術的実現可能性 | {評価} |
| コスト | {評価} |
| 差別化 | {評価} |
| 需要 | {評価} |
| 工数 | {評価} |

## ドキュメント

- 企画書: `02.Ideas/YYYY-MM-DD_{title}/overview.md`
- 壁打ち記録: `02.Ideas/YYYY-MM-DD_{title}/discussion.md`
- 参考資料: `02.Ideas/YYYY-MM-DD_{title}/references.md`
```

### Step 8: 完了報告

以下の情報を表示して完了とする:

1. **生成ファイル一覧**:
   - `02.Ideas/YYYY-MM-DD_{title}/overview.md`（企画書）
   - `02.Ideas/YYYY-MM-DD_{title}/discussion.md`（壁打ち記録）
   - `02.Ideas/YYYY-MM-DD_{title}/references.md`（参考資料）
2. **GitHub Issue URL**
3. **次のアクション**: 企画書の「ネクストアクション」セクションから最初のタスクを案内する

## 注意事項

- 全てのドキュメントは日本語で記述する
- 既存ファイルの上書きは禁止（同名ディレクトリが存在する場合はユーザーに確認する）
- 対話は急がない。ユーザーのペースに合わせる
- フィードバックは率直に、ただし建設的に行う
- 情報が足りない場合は「未定」として進め、後から埋めることを提案する
