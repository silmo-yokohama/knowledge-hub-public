# アイデア評価基準

PROFILE.md のスキルセットと制約条件に基づき、アイデアを以下の5軸で評価する。

## 評価軸

### 1. 技術的実現可能性

PROFILE.md に記載されたスキルセットで実現できるかを判断する。

| レベル | 説明 |
|--------|------|
| 容易 | メイン技術スタック（Next.js, Nuxt, TypeScript）で完結する |
| 可能 | 学習中の技術（Go, Python, AWS等）を含むが、調べながら対応可能 |
| 挑戦的 | 未経験の技術が必要だが、学習コストを投資する価値がある |
| 困難 | 専門知識（ML/データサイエンス、ネイティブアプリ等）が必要で、短期習得が難しい |

**判断のポイント**:
- フロントエンド中心なら「容易」寄り
- バックエンドやインフラが重い場合は「挑戦的」以上
- 「困難」でもAPI連携等で回避できる場合は1段階下げる

### 2. コスト

個人開発・予算なし前提で、どの程度のコストがかかるかを評価する。

| レベル | 説明 |
|--------|------|
| 無料 | 無料枠のみで運用可能（Vercel, Supabase Free, GitHub等） |
| 低コスト | 月額1,000円以下で運用可能 |
| 中コスト | 月額1,000〜5,000円程度かかる |
| 高コスト | 月額5,000円以上、または初期費用が必要 |

**判断のポイント**:
- ホスティング: Vercel/Cloudflare Pages の無料枠で収まるか
- DB: Supabase/PlanetScale/Turso の無料枠で収まるか
- API: 外部APIの無料枠制限を超えないか
- ドメイン: 独自ドメインが必要か

### 3. 差別化

既存サービスとの差別化ポイントがあるかを評価する。

| レベル | 説明 |
|--------|------|
| 高い | 明確な差別化要素がある、または既存サービスが存在しないニッチ |
| 中程度 | 既存サービスはあるが、特定のユースケースで優位性がある |
| 低い | 類似サービスが多数存在し、差別化が難しい |
| なし | 完全に車輪の再発明（ただし学習目的なら許容） |

**判断のポイント**:
- 「自分ならでは」の視点や切り口があるか
- ターゲットが十分に絞れているか
- 学習目的であれば差別化が低くても価値はある

### 4. 需要（Dogfooding）

自分自身が使いたいと思えるかを最重要視する。

| レベル | 説明 |
|--------|------|
| 高い | 自分が日常的に使いたい・今まさに欲しいと感じている |
| 中程度 | あれば便利だが、なくても困らない |
| 低い | 自分では使わないが、他の人には需要がありそう |
| なし | 誰が使うのか不明確 |

**判断のポイント**:
- 「自分が最初のユーザーになれるか」が最も重要
- 他人向けだけのサービスは個人開発のモチベーションが続かない
- 既に手動でやっている作業の自動化は需要が高い

### 5. 工数

MVP（最小限の動くプロダクト）の開発にかかる工数を見積もる。

| レベル | 目安期間 | 説明 |
|--------|----------|------|
| 小 | 1〜2日 | シンプルなツール・スクリプト・拡張機能 |
| 中 | 1〜2週間 | 基本機能を備えたWebアプリ |
| 大 | 2週間〜1ヶ月 | 複数機能・認証・DB連携を含むアプリ |
| 特大 | 1ヶ月以上 | 大規模アプリ（フェーズ分割を推奨） |

**判断のポイント**:
- 副業・空き時間での開発を前提とする
- 「特大」の場合はフェーズ分割でMVPを小さくできないか検討する
- 認証・決済などの共通基盤は工数に大きく影響する

## 総合評価の方針

- 5軸全てが高評価である必要はない
- **需要（Dogfooding）が「高い」なら、他の軸が多少弱くても前向きに評価する**
- **コストが「高コスト」の場合は、マネタイズ手段も併せて検討する**
- **工数が「特大」の場合は、MVP のスコープを絞る提案を行う**
- 学習目的のアイデアは、差別化が低くても技術的挑戦の価値で評価する

## フィードバックの姿勢

- 良い点は素直に認め、具体的に何が良いかを伝える
- 問題がある部分には率直に指摘する（曖昧にしない）
- 指摘には必ず代替案・改善案を添える
- 目的はアイデアを潰すことではなく、より良いアイデアにブラッシュアップすること
